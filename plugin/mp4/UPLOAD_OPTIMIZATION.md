# 云端上传优化配置指南

## 问题背景

当视频文件为 200-300MB 时，大批量上传到 MinIO/S3 可能出现部分文件上传失败的问题，主要原因：

1. **无并发控制**：大量录像同时结束时，瞬时并发上传过高，带宽需求远超网络容量
2. **超时配置不足**：默认 30 秒超时对 200MB+ 大文件严重不足
3. **重试机制缺陷**：上传失败后无法有效重试

## 环境参数

- **文件大小**：200-300MB
- **网络环境**：千兆网络（1Gbps = 125MB/s 理论带宽）
- **实际可用带宽**：约 80-100MB/s（考虑网络开销）
- **单文件理论上传时间**：2-3 秒
- **单文件实际上传时间**：5-15 秒（考虑 MinIO 处理、网络波动）

## 优化方案

### 1. 上传并发控制（必须配置）

在 MP4 plugin 配置中添加 `upload_concurrency` 参数：

```yaml
mp4:
  upload_concurrency: 3  # 最大并发上传数，200MB+ 文件建议 2-4
```

**配置说明：**
- 默认值：5
- **大文件（200MB+）建议范围：2-4**
- 根据 MinIO/S3 服务器性能和网络带宽调整
- 值越小越保守，值越大吞吐量越高但风险越大

**推荐配置（针对 200-300MB 文件）：**
- 小型部署（<50 路流）：2-3
- 中型部署（50-200 路流）：3-4
- 大型部署（>200 路流）：4-5

**带宽需求计算：**
```
所需带宽 = 并发数 × 文件大小 / 上传时间

示例（千兆网络）：
- 并发数 3，文件 250MB，上传时间 10s
- 带宽需求：3 × 250MB / 10s = 75MB/s = 600Mbps ✅ 可行
- 并发数 10，文件 250MB，上传时间 10s
- 带宽需求：10 × 250MB / 10s = 250MB/s = 2Gbps ❌ 超出千兆网络

### 2. S3 超时配置（推荐配置）

在 storage 配置中添加动态超时参数：

```yaml
storage:
  s3:
    timeout: 60s           # 基础超时（针对大文件调整）
    timeout_per_mb: 3s     # 每 MB 额外超时时间（千兆网络可以更短）
    max_timeout: 15m       # 最大超时限制（针对 200-300MB 文件）
```

**超时计算公式：**
```
实际超时 = min(基础超时 + 文件大小(MB) × 每MB超时, 最大超时)
```

**示例（针对 200-300MB 文件）：**
- 200MB 文件：60s + 200MB × 3s = 660s (11分钟)
- 250MB 文件：60s + 250MB × 3s = 810s (13.5分钟)
- 300MB 文件：60s + 300MB × 3s = 960s → 限制为 900s (15分钟)

**为什么这样配置：**
- 千兆网络下，200MB 文件理论上传时间：2-3 秒
- 实际上传时间（含 MinIO 处理）：5-15 秒
- 配置 11-15 分钟超时提供了 44-180 倍的安全余量
- 足以应对网络波动、MinIO 负载高峰等异常情况

**✅ 状态：已实现**（v5.x）

### 3. 完整配置示例

```yaml
# config.yaml
mp4:
  upload_concurrency: 5  # 并发控制

storage:
  type: s3
  s3:
    endpoint: "minio.example.com:9000"
    region: "us-east-1"
    access_key_id: "your-access-key"
    secret_access_key: "your-secret-key"
    bucket: "video-recordings"
    force_path_style: true
    use_ssl: false
    # 超时配置（动态计算）
    timeout: 60s
    timeout_per_mb: 3s
    max_timeout: 15m
    # Multipart 上传配置
    upload_part_size: 67108864  # 64MB
    upload_concurrency: 1       # 单文件并发分片数
```

## 优化效果

### 并发控制效果

**优化前：**
- 100 路流同时结束录像
- 100 个上传任务同时发起
- MinIO 服务过载，部分上传失败

**优化后：**
- 100 路流同时结束录像
- 最多 5 个上传任务并发执行
- 其余 95 个任务排队等待
- 上传成功率显著提升

### 超时优化效果

**优化前：**
- 10MB 文件，30 秒超时
- 网络速度 < 333KB/s 时超时失败

**优化后：**
- 10MB 文件，130 秒超时
- 网络速度 > 77KB/s 即可成功
- 容错能力提升 4.3 倍

## 监控和调优

### 日志监控

优化后会输出以下关键日志：

```
[MP4] upload concurrency initialized max_concurrent_uploads=5
[MP4] waiting for upload slot file=/path/to/video.mp4
[MP4] upload slot acquired file=/path/to/video.mp4
[S3] calculated timeout: 2m10s for file size 10 MB
[S3] uploading: bucket=videos key=stream/2024-01-01.mp4 size=10485760
[S3] upload successful: stream/2024-01-01.mp4
```

### 性能调优建议

1. **监控队列长度**
   - 如果经常看到 "waiting for upload slot"，考虑增加并发数
   - 如果上传失败率高，考虑减少并发数

2. **监控超时情况**
   - 如果大文件频繁超时，增加 `timeout_per_mb`
   - 如果小文件也超时，增加基础 `timeout`

3. **网络带宽评估**
   ```
   所需带宽 = 并发数 × 平均文件大小 / 平均上传时间

   示例：
   - 并发数：5
   - 平均文件大小：20MB
   - 期望上传时间：2分钟
   - 所需带宽：5 × 20MB / 120s = 0.83MB/s ≈ 6.6Mbps
   ```

## 故障排查

### 问题 1：上传仍然失败

**检查项：**
1. 确认 `upload_concurrency` 已配置且生效（查看启动日志）
2. 检查 MinIO/S3 服务器负载和网络连接
3. 增加超时时间或减少并发数

### 问题 2：上传速度太慢

**检查项：**
1. 当前并发数是否过低
2. 网络带宽是否充足
3. MinIO/S3 服务器性能是否足够

### 问题 3：队列积压严重

**检查项：**
1. 增加并发数（如果服务器能承受）
2. 优化录像分片大小（减少单个文件大小）
3. 考虑使用多个 MinIO 实例做负载均衡

## 进一步优化方向

1. **分片上传**：对于超大文件（>100MB），使用 S3 multipart upload
2. **优先级队列**：重要流的录像优先上传
3. **断点续传**：上传失败后支持从断点继续
4. **压缩上传**：在上传前压缩视频文件
5. **本地缓存**：上传失败时保留本地副本，定期重试
